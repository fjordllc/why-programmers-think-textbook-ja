# 付録 用語集

本文を読み進める途中で立ち止まりやすい言葉を、短くまとめる。厳密な定義を尽くすためではなく、読み直すときの足場にするための用語集である。

## [アジャイル](https://agilemanifesto.org/iso/ja/manifesto.html)

最初にすべてを決めきるより、短い単位で作って見せ、変化に対応しながら進める考え方の総称。第3章で触れた XP（Extreme Programming）やリファクタリングともつながる。

## [オープンソース](https://opensource.org/osd)

ソースコードを公開し、読む・直す・配ることを可能にする考え方や、その形で公開されたソフトウェア。第6章では「自由」の文脈で扱っている。

## [コメント](https://ja.wikipedia.org/wiki/%E3%82%B3%E3%83%A1%E3%83%B3%E3%83%88_(%E3%83%97%E3%83%AD%E3%82%B0%E3%83%A9%E3%83%A0))

コードの外側に書く説明文。本文では、何をしているかの説明をコメントに頼りすぎると、コードと食い違いやすいと述べた。理由や背景を書くときに強い。

## [CORBA（Common Object Request Broker Architecture）](https://www.omg.org/spec/CORBA/)

異なるコンピュータやプログラム同士を共通のやり方でつなごうとした、重厚な分散システムの標準。Object Management Group（OMG）が標準化を進めた。第9章では、Web の軽さと対比するための例として使っている。

## [GNU（GNU's Not Unix）](https://www.gnu.org/gnu/gnu-history.html)

リチャード・ストールマン（Richard Stallman）が始めた、誰でも使え、読み、直し、配れるソフトウェアを作ろうとするプロジェクト。第6章の中心。

## [HTTP（Hypertext Transfer Protocol）](https://httpwg.org/specs/rfc9110.html)

Web で文書やデータをやりとりするときの約束ごと。ティム・バーナーズ＝リー（Tim Berners-Lee）らが Web とともに形にしていった。ブラウザがサーバーに「これをください」と頼み、受け取るための基本の仕組み。

## [HTML（HyperText Markup Language）](https://html.spec.whatwg.org/)

Web ページの骨組みを表す書き方。ティム・バーナーズ＝リー（Tim Berners-Lee）が Web の文書のために考えた。見出し、段落、リンクなどを文書に印として埋め込む。

## [JavaScript](https://tc39.es/ecma262/)

Web ページに動きを与えるために広く使われるプログラミング言語。ブレンダン・アイク（Brendan Eich）が考案した。第9章では、Web の上に積み重なった要素の一つとして触れている。

## [KISS（Keep It Simple, Stupid）](https://en.wikipedia.org/wiki/KISS_principle)

できるだけ単純に保つべきだという考え方。

## [DRY（Don't Repeat Yourself）](https://en.wikipedia.org/wiki/Don%27t_repeat_yourself)

同じ知識やロジックを無意味に重複させない、という考え方。

## [REST（Representational State Transfer）](https://en.wikipedia.org/wiki/REST)

Web の軽さを生かした、比較的単純なやりとりの設計スタイル。ロイ・フィールディング（Roy Fielding）が博士論文の中で整理した。第9章では、重い分散標準と対比される。

## [Rails（Ruby on Rails）](https://rubyonrails.org/)

Ruby on Rails の略。デイヴィッド・ハイネマイヤー・ハンソン（David Heinemeier Hansson）が作った、Ruby で Web アプリケーションを作りやすくするためのフレームワーク。

## [リファクタリング](https://refactoring.com/)

外から見た動きを変えず、コードの内側の形だけを整えること。第3章で、作り直しとの違いを説明している。

## [レビュー](https://ja.wikipedia.org/wiki/%E3%82%B3%E3%83%BC%E3%83%89%E3%83%AC%E3%83%93%E3%83%A5%E3%83%BC)

書いたコードを別の人に見てもらい、理解や改善点を分かち合うこと。第5章では、属人化をほどくための文化として扱った。

## [YAGNI（You Aren't Gonna Need It）](https://en.wikipedia.org/wiki/You_aren%27t_gonna_need_it)

まだ必要になっていない機能を、先回りして作り込まないという考え方。
