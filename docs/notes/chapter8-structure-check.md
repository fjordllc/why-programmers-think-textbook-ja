# 第8章 構造チェック（執筆後の自己点検）

**拍の対応**

| 拍 | 本文の位置 |
|----|-----------|
| 事件 | 「自分で言語を作った人がいる」に驚く。なぜ車輪の再発明を／その言語を今あなたが書いているかも |
| ① 問題 | 言語は考え方を形づくる。難しいことはやらなくなり、やがて思いつかなくなる＝言語が許すことしか考えられない（檻の中から檻の外は見えない） |
| ② 最初の解決策 | ありものからいちばんマシなものを選ぶ（速い・流行・仕事で要る） |
| ③ 失敗 | どれを選んでも作者の価値観がついて回る。選んでいるつもりで選ばされている |
| ④ 新しい考え（崩し B・単声） | ある設計者が「書く人が楽しいこと」を第一に置いた（Ruby／Matz を匿名で）。道具は人間のためにある／機械を人間へ引き寄せる。他の設計者の価値観も併記し英雄化を回避 |
| ⑤ 今の当たり前 | 言語は性能の順位表でなく価値観の見本市。誤解「速い・流行で選ぶ」を内側で解く（問うべきは「自分は何を大事にしたいか」） |
| ⑥ まだ終わらない | どの価値観の言語が良いか＝開いた問い（第7章の枠組みを適用）。言語論争は終わらなくていい |
| 着地＋次章橋 | 「言語をつくるとは価値観で道具を選び直す自由」→「作っても届かない／世界へ送り出す壁」→ 第9章（Web）へ |

**所見**

- **崩し B（単声）の導出。** この章のエネルギーは歴史的事件でも読者の実技でもなく、一人の設計者の「確信」にある。だから単声で立てた。第7章の枠組み「開いた問い」を⑥でそのまま使い、「言語に優劣はつけられない＝価値観に優劣はつけられない」と接続。隣接は第7章 D・第9章 C（予定）なので B の重複なし。
- **①「言語の檻」を体感で書いた。** 抽象的な言語相対論（言語が思考を縛る）を、「難しい→やらない→思いつかない」「檻の中から外は見えない」という体感の段階で説明し、専門用語（サピア＝ウォーフ、Blub パラドックス）を出さずに伝えた。
- **Matz を匿名化し、Ruby 一色を回避。** FBC の母語が Ruby なので前面に出したい誘惑があるが、固有名を伏せ「ある設計者」とし、複数の価値観（多様なやり方／素直さ／読みやすさ＝Perl・Python 系の思想を匿名で）を併記。章の主役を「価値観で道具を選び直す」という考えに保った。要・編集判断：本書全体で Ruby/Matz の固有名をどこまで出すか（巻末注 or 本文）。
- **誤解の解き方。** 「速い・流行で選ぶ」を否定しきらず、「速さも一つの価値観」「流行の裏に多くの人の価値観への合致がある」と、STYLE_GUIDE「善悪を単純化しない」に沿って解いた。
- **「自由」抑制、八章連続で維持。** 本文は「選ばされている／選び直す」で進め、着地文だけで「自由」。

**要・出版前ファクト確認**
- 松本行弘（Matz）と Ruby（1995）、「プログラマーの幸福／楽しさ」を設計の中心に置いたという表現の出どころ
- 「一つのことに複数のやり方」＝Perl（Larry Wall, TMTOWTDI）、「素直さ・一つの明白なやり方」＝Python（Zen of Python）を匿名で示した妥当性
- 言語が思考を形づくるという主張の扱い（比喩として／言語相対論への目配り）
