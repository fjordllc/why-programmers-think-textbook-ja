# 第2章 構造チェック（執筆後の自己点検）

検証目的：(1) 第1章末「読めなさ」の橋を受けて第1部を一本につなげるか (2) 崩し D（実技）を、本書の code-light な質感を壊さずに入れられるか (3)「自由」抑制（着地文だけで落とす）を二章連続で再現できるか。

**拍の対応**

| 拍 | 本文の位置 |
|----|-----------|
| 事件 | レビューでロジックは無傷、変数名 `tmp`→`unpaidInvoices` だけ直された |
| ① 問題 | コードは書くより読まれる／半年後の自分は他人＝読めなさ（第1章から接続） |
| ② 最初の解決策 | コメントで全部説明する。「読みやすさ＝コメントの量」と信じた時代 |
| ③ 失敗 | 機械はコメントを読まない→腐って嘘をつく。間違ったコメントはない方がまし |
| ④ 新しい考え | コード自体を読めるものに。クヌース／SICP の一文／ファウラー。梃子は名前（コードの一部だから腐れない） |
| 崩し D | 同じ処理に三通りの名前（`f`／`getUsers`／`findDormantUsers`）、半年後に出会いたいのはどれか |
| ⑤ 今の当たり前 | コメント前に名前を疑う。誤解「コメントを書けば読みやすい」を内側で解く（何を=名前/コード、なぜ=コメント） |
| ⑥ まだ終わらない | よい名前とは何か（命名規約・英語/業務語・長短）＝開いた問い。向きだけは確定 |
| 着地＋次章橋 | 「名前をつけるとは…理解する自由を贈ることだ」→ 仕様変更へ（第3章へ） |

**所見**

- **「自由」抑制、二章連続で成立。** 本文は「読めなさ」「読める」「理解する」で進め、最後の着地文一文でだけ「自由」を出した。第1章と同じ呼吸を再現できた。
- **崩し D は list 形式で入れた。** 本書はここまでコードブロックゼロ。三通りの名前を `def` 等の構文ノイズなしの箇条書き＋インラインコードで提示し、prose 主体の質感を保ったまま「読者が選ぶ」実技にできた。コードブロックを立てるより本書に合う。
- **③失敗と⑤誤解の重複を回避。** ③は「コメントは機械が見ないから腐る（仕組みの失敗）」、⑤誤解は「だから貼れば読みやすい、ではない（運用の誤り）」と役割を分け、同じ話の二度打ちを避けた。
- **第1部が一本につながった。** 第1章末「複雑さを分けても読めなさが残る」を冒頭①で受け、第1部「複雑さと読めなさに抗う」を回収。章末は「記憶は保てても仕様は変わる」で第2部（変化）へ橋渡し。
- **崩しの導出（事前割り当てでない）。** この章のエネルギーは歴史的場面でなく「読者がいま味わう、名前だけ直されて納得いかない体験」にある。だから実技 D が自然。隣接の第1章は C なので種類も重ならない。
