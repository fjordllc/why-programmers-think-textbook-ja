# 第3章 構造チェック（執筆後の自己点検）

**拍の対応**

| 拍 | 本文の位置 |
|----|-----------|
| 事件 | 決めたはずの仕様が変わる→青ざめる新人／動じない先輩 |
| ① 問題 | 作る前に全部は決められない＝最初に固めると後で動けない |
| ② 最初の解決策 | 変わる前に決めきる。完璧な設計図／図からコード自動生成（CASE）／記法で固める |
| ③ 失敗 | 一度の仕様変更で図は古い絵に。図は設計でなく会話の道具だった |
| ④ 新しい考え（崩し B） | ケント・ベック「変化を抱きしめろ」を単声で立てる。カニンガム（技術的負債）・ファウラー（リファクタリング）が支え／象徴：短い宣言「計画より変化への対応を」 |
| ⑤ 今の当たり前 | 変わる前提で作る。誤解「リファクタリング＝作り直し」を内側で解く（動きを一切変えない手入れ） |
| ⑥ まだ終わらない | 設計をどこまで先に決めるか（行き当たりばったり↔決めすぎ）＝開いた問い |
| 着地＋次章橋 | 「設計とは最初にすべてを決めなくていい自由」→「直すたび壊していないと誰が保証するのか」→ テスト（第4章へ） |

**所見**

- **崩し B（単声）の導出。** この章のエネルギーは「変化は敵か前提か」という発想の反転にあり、その反転を一人の強い声＝ベック「Embrace Change」で担わせるのが最も効く。ただし英雄化を避けるため、カニンガム・ファウラー・宣言（17人）を支えに置き、「別々の現場で同じ結論に達した」と明示した。隣接は第2章 D・第4章 E なので B は重複しない。
- **③失敗＝失敗カタログの「CASE／UML 万能論」を採用。** 固有名（CASE・UML）は本文に出さず「図からコードを自動生成する道具」「記法」とぼかした。出典精度は巻末注で確保する方針（本書の年号・固有名は前景化しない方針に合わせた）。
- **誤解の解き方。** リファクタリング≠作り直しを、「動きを一ミリも変えない」という一点で切り分け、混同が火を噴く実害まで書いた。⑤の内側で解決（独立コーナーにしない）。
- **第2章からの接続と第4章への橋。** 「読める」（第2章）の次に「変えられる」（第3章）を置き、章末で「変えるたびに壊していないと誰が保証するのか」＝テスト（第4章）へ。既存の第4章が「触らないほうがいい」で開くので、橋が自然につながる。
- **「自由」抑制、三章連続で維持。** 本文は「変えられる」「動けない／動ける」で進め、着地文だけで「自由」を出した。

**要・出版前ファクト確認**
- CASE ツールの実態と帰結（1980〜90年代、図からのコード生成の不達）
- UML と「3 人の設計者（Booch／Rumbaugh／Jacobson）」、モデル駆動の万能論
- ケント・ベック『Extreme Programming Explained: Embrace Change』の "Embrace Change" 表現
- ウォード・カニンガムの「技術的負債（technical debt）」メタファの初出（1992 OOPSLA 体験報告）
- ファウラーのリファクタリング定義「外部から見た振る舞いを変えずに内部構造を改善する」
- アジャイルソフトウェア開発宣言（2001・Snowbird・17 名）「計画に従うことよりも変化への対応を」
