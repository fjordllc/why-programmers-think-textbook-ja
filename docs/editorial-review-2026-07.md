# 技術書編集レビューと改稿方針（2026-07）

## 初回レビュー要点

### 1. 受講生向けの足場が不足
- 歴史読み物としては強いが、初学者が立ち止まったときに戻る先が年表しかない
- 用語集、人物の短い説明、参考文献、追加で読む導線が必要

### 2. 匿名化が学習上の損失を生んでいる箇所がある
- 第8章は Ruby を知る読者だけが得をする構造になっていた
- 第9章も CORBA / HTML / HTTP / REST などの名札が薄く、概念の輪郭が掴みにくい

### 3. 断言が強く、初学者に規範として誤学習されうる
- 第2章の「名前は本体だから、腐れない」
- 第4章の「テストが守っているのは、品質ではない」
- 方向としては正しいが、教材としては少し温度を下げたほうが安全

### 4. 第6章の現代接続をもう一段増やしたい
- 思想史としては通っている
- ただし、受講生にとっての実感との橋がやや薄い

### 5. 出典へ戻る導線が必要
- 年表だけでは資料性が足りない
- 最低限の参考文献一覧を置きたい

## 今回の改稿方針

- 第2章: コメントと名前の関係の断言を少し緩める
- 第4章: テストと品質の関係を「第一義ではない」に調整する
- 第6章: OSS を受講生の日常へつなぐ具体例を加える
- 第8章: Ruby / まつもとゆきひろを本文で明示する
- 第9章: CORBA / HTML / HTTP / JavaScript / Rails / REST を本文で名指す
- 付録追加:
  - 用語集
  - 参考文献

## 改稿後に確認したい点

- 補助線の追加で、本文の読み物としての軽さを壊していないか
- 固有名詞の追加で、資料集っぽさが強くなりすぎていないか
- 付録が「読むための足場」として十分に機能しているか
